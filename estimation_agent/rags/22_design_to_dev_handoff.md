# デザイン→開発引き継ぎガイド

## 引き継ぎのタイミング

### 理想的なフロー
1. デザイン完成
2. **開発者レビュー**（実装可能性チェック）← 重要！
3. 必要に応じて調整
4. 開発開始

### なぜ開発者レビューが重要か
- 技術的に実装困難なデザインを早期発見
- パフォーマンス問題の回避
- 開発工数の正確な見積もり

## 開発者レビューで確認すべきこと

### 技術的実装可能性
- [ ] アニメーションの実装難易度
- [ ] レスポンシブ対応の複雑さ
- [ ] 使用フォントのライセンス・Webフォント対応
- [ ] 画像の最適化必要性
- [ ] ブラウザ互換性

### パフォーマンス
- [ ] 画像サイズ・数（ページ読み込み速度）
- [ ] アニメーションの重さ
- [ ] モバイル表示の最適化
- [ ] 初期表示速度

### アクセシビリティ
- [ ] コントラスト比（WCAG 2.1準拠）
- [ ] フォントサイズ（最小14px推奨）
- [ ] キーボード操作対応
- [ ] スクリーンリーダー対応

## 引き継ぎ時の成果物

### デザイン会社から
1. **Figmaファイル**（閲覧権限）
2. **デザインガイド**（PDF）
3. **アセット一式**（ZIP）
   - 画像（PNG, SVG, WebP）
   - アイコン
   - ロゴ
4. **フォント一覧**（ライセンス情報含む）
5. **プロトタイプ**（画面遷移確認用）

### 営業から開発者へ
1. **プロジェクト概要**
2. **要件定義書**
3. **ワイヤーフレーム**
4. **デザインファイル**
5. **スケジュール**
6. **予算**

## よくある問題と対策

### 問題1: デザインが実装困難
**症状**: 複雑なアニメーション、特殊なレイアウト  
**対策**: デザイン完成前に開発者レビュー

### 問題2: フォントが商用利用不可
**症状**: 使用フォントのライセンス問題  
**対策**: デザイン依頼時にWebフォント使用を明記

### 問題3: アセットが不足
**症状**: 必要な画像・アイコンがない  
**対策**: 引き継ぎチェックリストを使用

### 問題4: レスポンシブ仕様が不明確
**症状**: タブレット・スマホ表示が未定義  
**対策**: 全ブレークポイントのデザイン確認

## 引き継ぎミーティングの進め方

### アジェンダ（60分）
1. **プロジェクト概要説明**（10分）
   - 目的・背景
   - ターゲットユーザー
   - スケジュール

2. **デザインコンセプト説明**（10分）
   - デザインの方向性
   - ブランドアイデンティティ
   - 参考サイト

3. **Figmaファイルウォークスルー**（20分）
   - 主要画面の説明
   - コンポーネントの使い方
   - レスポンシブ対応
   - インタラクション

4. **技術的な質疑応答**（15分）
   - 実装上の懸念点
   - パフォーマンス考慮事項
   - ブラウザ対応

5. **スケジュール確認**（5分）
   - 開発期間
   - マイルストーン
   - レビュータイミング

### 参加者
- 営業担当
- デザイン会社担当者
- フロントエンド開発者（必須）
- バックエンド開発者（必要に応じて）
- プロジェクトマネージャー

## 開発開始後のデザイン修正

### 原則
- 開発開始後の大幅な修正は避ける
- やむを得ない場合は、影響範囲と工数を見積もる

### 修正が必要な場合
1. 影響範囲の確認
2. 追加工数の見積もり
3. スケジュールへの影響確認
4. クライアント承認
5. 修正実施
